The Flexible Software Package (FSP) is organized as a set of layered software components that sit between your application and the RA MCU hardware. Each layer has a well-defined responsibility, and layers communicate through standardized C interfaces defined in API header files.

## Layer overview

```
┌─────────────────────────────────────────────────────┐
│                  Your Application                   │
├─────────────────────────────────────────────────────┤
│         Middleware  (rm_* modules)                  │
│   RTOS · Networking · Security · Motor Control      │
├─────────────────────────────────────────────────────┤
│          HAL Drivers  (r_* modules)                 │
│   UART · SPI · I2C · ADC · Timer · Flash · …        │
├─────────────────────────────────────────────────────┤
│       Board Support Package  (BSP)                  │
│   Startup · Clocks · Pins · CMSIS · Interrupts      │
├─────────────────────────────────────────────────────┤
│              RA MCU Hardware                        │
└─────────────────────────────────────────────────────┘
```

| Layer | Prefix | Location | Purpose |
|---|---|---|---|
| BSP | `bsp_*` | `ra/fsp/src/bsp/` | Startup, clocks, pins, CMSIS |
| HAL Drivers | `r_*` | `ra/fsp/src/r_*/` | Peripheral driver implementations |
| Middleware | `rm_*` | `ra/fsp/src/rm_*/` | Stacks and abstractions above HAL |
| Application | — | Your project | Business logic |

## Interface-based design

Every FSP module is defined by an **interface** — a C header in `ra/fsp/inc/api/` that declares:

- A configuration struct (`*_cfg_t`) holding all compile-time settings.
- A control block type (`*_ctrl_t`) that holds the driver's runtime state.
- An API struct (`*_api_t`) containing function pointers for every operation.
- An instance struct (`*_instance_t`) bundling the three types above.

Concrete driver implementations live under `ra/fsp/src/` and fill in those function pointers. Your application code only depends on the interface header, so you can swap one implementation for another — for example, replacing `r_sci_uart` with `r_uarta` — without touching any application logic.

```c
/* Interface contract (inc/api/r_uart_api.h) */
typedef struct st_uart_api
{
    fsp_err_t (* open)(uart_ctrl_t * const p_ctrl, uart_cfg_t const * const p_cfg);
    fsp_err_t (* read)(uart_ctrl_t * const p_ctrl, uint8_t * const p_dest, uint32_t const bytes);
    fsp_err_t (* write)(uart_ctrl_t * const p_ctrl, uint8_t const * const p_src, uint32_t const bytes);
    fsp_err_t (* close)(uart_ctrl_t * const p_ctrl);
    /* … additional operations … */
} uart_api_t;

/* Concrete instance (generated by RASC, src/r_sci_uart/) */
extern const uart_instance_t g_uart0;

/* Application code — depends only on the interface */
g_uart0.p_api->open(g_uart0.p_ctrl, g_uart0.p_cfg);
```

## Module naming conventions

FSP uses a consistent prefix scheme to signal a module's role at a glance.

| Prefix | Role | Example |
|---|---|---|
| `r_` | HAL peripheral driver | `r_uart`, `r_spi`, `r_adc`, `r_timer`, `r_flash` |
| `rm_` | Middleware or abstraction layer | `rm_comms`, `rm_motor`, `rm_wifi`, `rm_touch` |
| `bsp_` | Board Support Package | `bsp_clocks`, `bsp_io`, `bsp_irq` |

API header files mirror this naming: `r_uart_api.h`, `r_timer_api.h`, `rm_comms_api.h`, and so on. Each header is self-contained and includes only what it needs from lower layers.

## Source tree layout

```
ra/fsp/
├── inc/
│   ├── api/          ← Interface headers (r_*_api.h, rm_*_api.h, bsp_api.h)
│   └── fsp_features.h
└── src/
    ├── bsp/
    │   ├── cmsis/    ← ARM CMSIS-CORE device files
    │   └── mcu/
    │       ├── all/  ← Shared BSP code (clocks, IRQ, IO, security)
    │       ├── ra6m5/
    │       └── …     ← One subdirectory per supported MCU family
    ├── r_sci_uart/   ← HAL driver implementation
    ├── r_gpt/        ← HAL driver implementation
    └── rm_comms_i2c/ ← Middleware implementation
```

## Module selection with RASC

You select and configure FSP modules using the **RA Smart Configurator (RASC)**, which is embedded in e² studio and available as a standalone tool for IAR and Keil MDK.

<Steps>
  <Step title="Open the FSP configuration">
    Double-click `configuration.xml` in your project. RASC opens the Stacks view.
  </Step>
  <Step title="Add a module stack">
    Click **New Stack** and choose a HAL driver or middleware component from the catalog. RASC resolves dependencies and adds required sub-modules automatically.
  </Step>
  <Step title="Configure properties">
    Select a module in the Stacks view and edit its properties in the panel below. RASC validates settings and flags conflicts.
  </Step>
  <Step title="Generate code">
    Click **Generate Project Content**. RASC writes `ra_gen/` source files — including the `g_*` instance variables — into your project. Do not edit these files manually.
  </Step>
</Steps>

<Note>
Generated files in `ra_gen/` are overwritten on every code-generation run. Place custom initialization logic in `src/hal_entry.c` or your own source files instead.
</Note>
