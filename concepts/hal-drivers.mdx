Hardware Abstraction Layer (HAL) drivers in FSP provide a consistent, interface-driven way to use RA MCU peripherals. Every driver follows the same structural pattern, so switching between peripherals — or swapping one driver implementation for another — requires minimal code changes.

## The interface pattern

Each HAL driver is defined by three types, all declared in a single API header under `ra/fsp/inc/api/`:

| Type | Purpose | Example |
|---|---|---|
| `*_cfg_t` | Static configuration set before `open()` | `uart_cfg_t`, `timer_cfg_t` |
| `*_ctrl_t` | Opaque runtime control block (allocated by you) | `uart_ctrl_t`, `timer_ctrl_t` |
| `*_api_t` | Struct of function pointers — the interface contract | `uart_api_t`, `timer_api_t` |

These three are bundled into a fourth type, `*_instance_t`, which is what RASC generates as a named global variable in your project:

```c
typedef struct st_uart_instance
{
    uart_ctrl_t      * p_ctrl;   ///< Pointer to the control block
    uart_cfg_t const * p_cfg;    ///< Pointer to the configuration
    uart_api_t const * p_api;    ///< Pointer to the function-pointer table
} uart_instance_t;
```

Your application interacts with the driver exclusively through the instance:

```c
/* RASC generates this in ra_gen/hal_data.h and ra_gen/hal_data.c */
extern const uart_instance_t g_uart0;

/* Your code calls through the function pointer table */
g_uart0.p_api->open(g_uart0.p_ctrl, g_uart0.p_cfg);
```

## The UART interface in detail

`r_uart_api.h` is one of the most complete API headers and illustrates the full pattern.

### Configuration struct

```c
typedef struct st_uart_cfg
{
    uint8_t          channel;       ///< Hardware channel number
    uart_data_bits_t data_bits;     ///< 7, 8, or 9 data bits
    uart_parity_t    parity;        ///< None, even, odd, or zero
    uart_stop_bits_t stop_bits;     ///< 1 or 2 stop bits
    uint8_t          rxi_ipl;       ///< Receive interrupt priority
    IRQn_Type        rxi_irq;       ///< Receive interrupt number
    uint8_t          txi_ipl;       ///< Transmit interrupt priority
    IRQn_Type        txi_irq;       ///< Transmit interrupt number
    uint8_t          tei_ipl;       ///< Transmit-end interrupt priority
    IRQn_Type        tei_irq;       ///< Transmit-end interrupt number
    uint8_t          eri_ipl;       ///< Error interrupt priority
    IRQn_Type        eri_irq;       ///< Error interrupt number
    transfer_instance_t const * p_transfer_rx; ///< DMA/DTC for RX, or NULL
    transfer_instance_t const * p_transfer_tx; ///< DMA/DTC for TX, or NULL
    void (* p_callback)(uart_callback_args_t * p_args); ///< Event callback
    void       * p_context;         ///< User data passed to the callback
    void const * p_extend;          ///< Implementation-specific extension
} uart_cfg_t;
```

### Control block

The control block type is declared as `void uart_ctrl_t`. The actual size is private to the driver implementation. RASC generates a correctly-sized concrete type and allocates it for you:

```c
/* Generated by RASC in ra_gen/hal_data.c */
sci_uart_instance_ctrl_t g_uart0_ctrl;
```

You never inspect or modify the control block directly.

### API struct (function pointer table)

```c
typedef struct st_uart_api
{
    fsp_err_t (* open)(uart_ctrl_t * const p_ctrl, uart_cfg_t const * const p_cfg);
    fsp_err_t (* read)(uart_ctrl_t * const p_ctrl, uint8_t * const p_dest, uint32_t const bytes);
    fsp_err_t (* write)(uart_ctrl_t * const p_ctrl, uint8_t const * const p_src, uint32_t const bytes);
    fsp_err_t (* baudSet)(uart_ctrl_t * const p_ctrl, void const * const p_baudrate_info);
    fsp_err_t (* infoGet)(uart_ctrl_t * const p_ctrl, uart_info_t * const p_info);
    fsp_err_t (* communicationAbort)(uart_ctrl_t * const p_ctrl, uart_dir_t communication_to_abort);
    fsp_err_t (* callbackSet)(uart_ctrl_t * const p_ctrl,
                              void (* p_callback)(uart_callback_args_t *),
                              void * const p_context,
                              uart_callback_args_t * const p_callback_memory);
    fsp_err_t (* close)(uart_ctrl_t * const p_ctrl);
    fsp_err_t (* readStop)(uart_ctrl_t * const p_ctrl, uint32_t * remaining_bytes);
    fsp_err_t (* receiveSuspend)(uart_ctrl_t * const p_ctrl);
    fsp_err_t (* receiveResume)(uart_ctrl_t * const p_ctrl);
} uart_api_t;
```

## Peripheral lifecycle: open → use → close

Every HAL driver follows the same lifecycle. Calling functions out of order returns `FSP_ERR_NOT_OPEN` or `FSP_ERR_ALREADY_OPEN`.

```
open()  ──►  read() / write() / other operations  ──►  close()
  │                                                         │
  │ Allocates hardware resource,                            │ Releases hardware,
  │ applies configuration,                                  │ disables clocks,
  │ enables interrupts.                                     │ clears control block.
```

<Steps>
  <Step title="Open the peripheral">
    Call `open()` with the control block and configuration. This configures the hardware registers, enables the peripheral clock, and registers interrupt vectors.

    ```c
    fsp_err_t err = g_uart0.p_api->open(g_uart0.p_ctrl, g_uart0.p_cfg);
    FSP_ERROR_RETURN(FSP_SUCCESS == err, err);
    ```
  </Step>
  <Step title="Use the peripheral">
    Call the driver's operational functions. For UART, this means `write()` for transmission and `read()` for reception. Results and error conditions are delivered through the callback registered in `uart_cfg_t`.

    ```c
    uint8_t message[] = "Hello, RA!\r\n";
    err = g_uart0.p_api->write(g_uart0.p_ctrl, message, sizeof(message) - 1U);
    FSP_ERROR_RETURN(FSP_SUCCESS == err, err);
    ```
  </Step>
  <Step title="Handle events in the callback">
    The driver calls your registered callback from an ISR when a transfer completes or an error occurs. Use `p_args->event` to distinguish events.

    ```c
    void uart0_callback(uart_callback_args_t * p_args)
    {
        switch (p_args->event)
        {
            case UART_EVENT_TX_COMPLETE:
                /* Transmission finished — safe to reuse the buffer */
                break;
            case UART_EVENT_RX_COMPLETE:
                /* Reception buffer is full */
                break;
            case UART_EVENT_ERR_FRAMING:
            case UART_EVENT_ERR_PARITY:
            case UART_EVENT_ERR_OVERFLOW:
                /* Handle communication error */
                break;
            default:
                break;
        }
    }
    ```
  </Step>
  <Step title="Close the peripheral">
    Call `close()` when the peripheral is no longer needed. This releases the hardware resource and stops the peripheral clock, reducing power consumption.

    ```c
    g_uart0.p_api->close(g_uart0.p_ctrl);
    ```
  </Step>
</Steps>

## Error handling with `fsp_err_t`

All API functions return `fsp_err_t`. `FSP_SUCCESS` (0) means success. Any other value indicates a specific error.

```c
fsp_err_t err = g_uart0.p_api->write(g_uart0.p_ctrl, p_data, length);

switch (err)
{
    case FSP_SUCCESS:
        break;
    case FSP_ERR_NOT_OPEN:
        /* open() was not called first */
        break;
    case FSP_ERR_IN_USE:
        /* A previous transfer is still in progress */
        break;
    case FSP_ERR_INVALID_ARGUMENT:
        /* length is zero, or p_data is NULL */
        break;
    default:
        /* Unexpected error — log and recover */
        break;
}
```

FSP provides a convenience macro for early-exit error propagation:

```c
/* Returns err from the enclosing function if the condition is false */
FSP_ERROR_RETURN(FSP_SUCCESS == err, err);
```

<Note>
Peripheral-specific error codes occupy separate numeric ranges in `fsp_common_api.h`. UART errors start at 200, SPI at 300, clock (CGC) errors at 400, and Flash errors at 500. This makes it easy to identify the source of an error at a glance.
</Note>

## Available HAL driver interfaces

FSP includes HAL driver interfaces for the full range of RA peripherals:

<Tabs>
  <Tab title="Communication">
    | Header | Peripheral |
    |---|---|
    | `r_uart_api.h` | UART / USART |
    | `r_spi_api.h` | SPI master/slave |
    | `r_i2c_master_api.h` | I2C master |
    | `r_i2c_slave_api.h` | I2C slave |
    | `r_i3c_api.h` | I3C |
    | `r_can_api.h` | CAN |
    | `r_lin_api.h` | LIN |
    | `r_ether_api.h` | Ethernet (ETHERC/EDMAC) |
  </Tab>
  <Tab title="Timers and PWM">
    | Header | Peripheral |
    |---|---|
    | `r_timer_api.h` | General-purpose timer (GPT, AGT) |
    | `r_capture_api.h` | Input capture |
    | `r_i2s_api.h` | I2S / audio serial |
  </Tab>
  <Tab title="Analog">
    | Header | Peripheral |
    |---|---|
    | `r_adc_api.h` | ADC |
    | `r_dac_api.h` | DAC |
    | `r_comparator_api.h` | Analog comparator |
  </Tab>
  <Tab title="Storage and security">
    | Header | Peripheral |
    |---|---|
    | `r_flash_api.h` | Code flash and data flash |
    | `r_sdhi_api.h` | SD/MMC host interface |
    | `r_crc_api.h` | CRC calculator |
  </Tab>
</Tabs>
