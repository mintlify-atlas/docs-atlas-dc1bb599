The Board Support Package (BSP) is the lowest software layer in FSP. It runs before `main()`, brings the MCU to a known operational state, and exposes a set of APIs that all other FSP modules rely on.

## What the BSP does

<CardGroup cols={2}>
  <Card title="Startup and reset" icon="power-off">
    Executes the reset handler, initializes the stack pointer, copies initialized data to RAM, and clears the BSS segment before branching to `main()`.
  </Card>
  <Card title="Clock initialization" icon="clock">
    Configures the system clock tree — PLL, sub-clocks, peripheral clock dividers — to the frequencies you specify in RASC.
  </Card>
  <Card title="Pin configuration" icon="plug">
    Applies the IOPORT pin multiplexing and drive-strength settings generated by the RASC pin configurator.
  </Card>
  <Card title="CMSIS and MCU headers" icon="file-code">
    Pulls in CMSIS-CORE device files, IRQ vector tables, and MCU-specific feature definitions for the selected device.
  </Card>
</CardGroup>

A shared `all/` directory contains code that applies to every RA device, including clock management (`bsp_clocks.h`), IRQ handling (`bsp_irq.h`), I/O port control (`bsp_io.h`), register protection (`bsp_register_protection.h`), and security configuration (`bsp_security.h`).

## The `bsp_api.h` umbrella header

All FSP modules include a single BSP header:

```c
#include "bsp_api.h"
```

`bsp_api.h` is an umbrella header that pulls in everything a driver needs from the BSP:

| Included file | Provides |
|---|---|
| `fsp_common_api.h` | `fsp_err_t` error codes, `FSP_HEADER`/`FSP_FOOTER` macros |
| `bsp_cfg.h` | Project-specific MCU and board configuration |
| `renesas.h` / `system.h` | CMSIS-CORE device register definitions |
| `bsp_clocks.h` | Clock tree configuration and frequency APIs |
| `bsp_irq.h` | IRQ enable/disable and vector table management |
| `bsp_io.h` | IOPORT pin read/write and drive-strength control |
| `bsp_register_protection.h` | Protected register write/unlock helpers |
| `bsp_security.h` | TrustZone and security attribution unit (SAU) setup |
| `bsp_delay.h` | Software delay functions (`R_BSP_SoftwareDelay`) |
| `bsp_mcu_api.h` | `R_FSP_VersionGet` and MCU identification |

You never need to include individual BSP headers — `bsp_api.h` handles all transitive includes.

## FSP error codes

Every FSP function returns an `fsp_err_t` value. The error codes are defined in `fsp_common_api.h` and shared across all modules.

```c
typedef enum e_fsp_err
{
    FSP_SUCCESS             = 0,   ///< Operation completed successfully
    FSP_ERR_ASSERTION       = 1,   ///< A critical assertion has failed
    FSP_ERR_INVALID_POINTER = 2,   ///< Pointer points to invalid memory location
    FSP_ERR_INVALID_ARGUMENT = 3,  ///< Invalid input parameter
    FSP_ERR_INVALID_CHANNEL = 4,   ///< Selected channel does not exist
    FSP_ERR_INVALID_MODE    = 5,   ///< Unsupported or incorrect mode
    FSP_ERR_NOT_OPEN        = 7,   ///< API not open — call open() first
    FSP_ERR_ALREADY_OPEN    = 14,  ///< Channel already open in a different configuration
    FSP_ERR_TIMEOUT         = 20,  ///< Operation timed out
    /* … peripheral-specific ranges … */
} fsp_err_t;
```

The convention is straightforward: `FSP_SUCCESS` (0) means success; any non-zero value is an error. Peripheral-specific errors start at higher offsets (e.g., UART errors at 200, SPI errors at 300, clock errors at 400).

Always check return values in production code:

```c
fsp_err_t err = g_uart0.p_api->open(g_uart0.p_ctrl, g_uart0.p_cfg);
if (FSP_SUCCESS != err)
{
    /* Handle the error — err tells you exactly what went wrong */
}
```

<Tip>
During development, define `FSP_CFG_ASSERT` to enable internal assertion checks. This catches misuse early at the cost of a small code-size increase.
</Tip>

## Selecting the BSP in RASC

<Steps>
  <Step title="Create or open a project">
    In e² studio, choose **File → New → C/C++ Project** and select **Renesas RA** in the left pane. If you are opening an existing project, skip to the next step.
  </Step>
  <Step title="Select your MCU">
    In the **New RA C/C++ Project** wizard, choose your device part number (for example, `R7FA6M5BH3CFB`). RASC automatically selects the matching BSP directory under `ra/fsp/src/bsp/mcu/`.
  </Step>
  <Step title="Configure clocks">
    Open **Clocks** in the RASC configurator. Set the system clock source (HOCO, MOCO, PLL, external oscillator) and the dividers for ICLK, PCLKA–D, FCLK, and BCLK. RASC validates that no clock exceeds its rated maximum.
  </Step>
  <Step title="Configure pins">
    Open **Pins** in the RASC configurator. Assign peripheral signals to physical pins and set drive strength, pull-up, and input-enable options. RASC prevents conflicts between peripherals sharing the same pin.
  </Step>
  <Step title="Generate code">
    Click **Generate Project Content**. RASC writes BSP configuration to `ra_cfg/fsp_cfg/bsp/bsp_cfg.h` and pin settings to `ra_gen/common_data.c`.
  </Step>
</Steps>

<Warning>
Changing the MCU part number after initial project creation requires regenerating all BSP configuration. Verify clock and pin assignments after any device change.
</Warning>
